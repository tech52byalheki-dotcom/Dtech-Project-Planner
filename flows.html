Pipeline — click any step to inspect

1
Customer browses products
Paginated, filtered product catalogue fetched from MySQL via dtechmobileapp
GET /customer/v1/products
dtechmobileapp MySQL read
2
Add item to cart
Cart stored in Redis. Includes product_id, vendor_id, inventory_id, quantity.
POST /customer/v1/product/add-to-cart
dtechmobileapp Redis write
3
Create Razorpay order
Backend calls Razorpay API, returns order_id + key to mobile app
POST /payment/v1/order/create
dtechmobileapp Razorpay API
4
Razorpay payment UI
Native Razorpay payment sheet opens on customer's mobile device
Mobile SDK
5
Capture & verify payment
Signature verified → CustomerOrder + Items written to MySQL → status CONFIRMED
POST /payment/v1/order/payments/capture
dtechmobileapp MySQL write
6
Razorpay webhook confirmation
Async event from Razorpay updates payment_status = PAID in MySQL
POST /payment/v1/order/payments/webhook/capture
Webhook MySQL write
7
Admin views customer orders
Admin dashboard reads customer_orders from shared MySQL via dtechcommonserver
GET /admin/v1/usersorders/users/orders
dtechcommonserver MySQL read
8
Admin assigns driver
Driver linked to order → Expo push notification sent to driver's device
POST /admin/v1/usersorders/orders/:orderId/assign-driver
dtechcommonserver Expo Push
9
Driver accepts & picks up
Driver accepts order then marks it as picked up. Push notification sent to customer at each step.
POST /driver/v1/:id/accept → PUT /driver/v1/:id/pickup
dtechcommonserver Push → Customer
10
Order delivered
Driver marks delivered → order_status = DELIVERED → customer sees update in app
PUT /driver/v1/:id/deliver
dtechcommonserver MySQL write

Pipeline — click any step to inspect

1
Vendor logs in
OTP-based or password login. userType: "vendor". Returns JWT.
POST /common/v1/validateLogin
dtechcommonserver
2
View available inventory & products
Vendor browses products available from warehouses for restocking
GET /vendor/v1/inventory/products
dtechcommonserver MySQL read
3
Create product order
Creates order_details (status: pending) + order_products rows. Snapshots product name, price, SKU at order time.
POST /vendor/v1/create-product-order
dtechcommonserver MySQL write
4
Admin reviews incoming orders
Admin dashboard fetches all pending vendor orders
GET /admin/v1/orders/incoming
dtechcommonserver MySQL read
5
Admin approves order
order_status → approved. Inventory quantity updated at the warehouse. Optionally sets pickup details.
POST /admin/v1/orders/approve (or /newapprove)
dtechcommonserver MySQL write
6
Vendor generates PDF manifest
Puppeteer renders order as PDF. Returned as binary to vendor for printing / filing.
POST /vendor/v1/order/generate-manifest
dtechcommonserver Puppeteer PDF

Backend

DB tables
drivers

Pipeline — click any step to inspect

1
Driver submits registration & documents
Multipart upload: driving license, vehicle registration, profile photo. Files stored in Modules/Driver/Uploads/.
POST /common/v1/driver/update-profile
dtechcommonserver MySQL write
2
Admin reviews driver profile & documents
Admin views full driver profile including uploaded documents via DocumentViewer component in dashboard
GET /admin/v1/drivers/:driverId
dtechcommonserver MySQL read
3
Admin approves or rejects
On approval: isDocumentVerified = true. On rejection: reason stored as remarks.
PUT /admin/v1/drivers/:driverId/verify (or /reject)
dtechcommonserver MySQL write
4
Driver active — can receive orders
Verified driver logs in, sets status to online, and starts appearing in admin's available drivers list for order assignment.
PUT /driver/v1/status → GET /admin/v1/usersorders/drivers/available
dtechcommonserver
Steps
9
Actors
Driver · Admin · Customer
Backend
dtechcommonserver
External
Expo Push
DB tables
customer_orders · drivers

Pipeline — click any step to inspect

1
Driver registers push token
Driver saves Expo push token so the server can send order assignment notifications to their device.
POST /driver/v1/save-push-token
dtechcommonserver Redis write
2
Driver goes online
Driver sets status to online — becomes visible in admin's available drivers list for order assignment.
PUT /driver/v1/status
dtechcommonserver MySQL write
3
Admin assigns order to driver
Admin selects a verified online driver from the available list and assigns them to a confirmed customer order. Push notification fired immediately.
POST /admin/v1/usersorders/orders/:orderId/assign-driver
dtechcommonserver Push → Driver
4
Driver views order details
Driver opens the order to see customer address, items, and delivery instructions before accepting.
GET /driver/v1/:order_id
dtechcommonserver MySQL read
5
Driver accepts the order
Driver confirms they will fulfil this delivery. Customer receives "Driver accepted your order" push notification.
POST /driver/v1/:order_id/accept
dtechcommonserver Push → Customer
6
Driver updates GPS location
Driver app continuously updates coordinates while en route. Used for live tracking in the customer app.
PUT /driver/v1/location
dtechcommonserver MySQL write
7
Driver marks order as picked up
Driver confirms they have collected the package from the vendor. Customer receives "Your order is on the way" notification.
PUT /driver/v1/:order_id/pickup
dtechcommonserver Push → Customer
8
Driver delivers — order complete
Driver marks delivered at the customer's address. Order status → DELIVERED. Customer notified. Driver earnings and delivery count updated.
PUT /driver/v1/:order_id/deliver
dtechcommonserver Push → Customer
9
Alternate path — Cancel or RTO
If delivery fails: driver can cancel the order or initiate Return to Origin (RTO). Both update order status and trigger admin visibility.
POST /driver/v1/:order_id/cancel · POST /driver/v1/:order_id/return
dtechcommonserver MySQL write

DB tables
vendors

Pipeline — click any step to inspect

1
Vendor submits registration & documents
Multipart upload: business details, GST ID, GST file, trade licence file. Stored via Multer middleware.
POST /common/v1/vendor/update-profile
dtechcommonserver MySQL write
2
Admin lists vendors pending verification
Admin dashboard shows vendors where isDocumentVerified = false. Filterable by city, status, search.
GET /admin/v1/vendors
dtechcommonserver MySQL read
3
Admin reviews full vendor profile & documents
Full profile including gstFile, tradeLicenseFile URLs. Admin views documents via DocumentViewer component in dashboard.
GET /admin/v1/vendors/:vendorId
dtechcommonserver MySQL read
4
Admin approves or rejects vendor documents
On approval: isDocumentVerified = true, isActive = true. On rejection: remarks stored. Admin can also assign warehouse.
POST /admin/v1/vendors/verify · PATCH /admin/v1/vendors/:id/status
dtechcommonserver MySQL write
5
Vendor logs in and becomes operational
Verified vendor can now authenticate, place restock orders, manage inventory, raise support tickets, and generate PDF manifests.
POST /common/v1/validateLogin (userType: "vendor")
dtechcommonserver
